# i18n Design Spec ‚Äî money-flow

**Date:** 2026-05-27  
**Status:** Approved  

## Summary

Add multi-language support (English + Vietnamese, extensible) to both the web (Next.js 16) and mobile (Expo) apps using `react-i18next`. Translation files live in a new shared `packages/i18n` package. Language preference is stored locally per device and synced to the user's profile on the API.

---

## 1. Shared Package ‚Äî `packages/i18n`

New internal package `@repo/i18n`.

### File structure

```
packages/i18n/
  src/
    locales/
      en.json                        # source of truth ‚Äî all keys defined here
      vi.json                        # Vietnamese translations
    index.ts                         # exports: createI18n(), supportedLocales, defaultLocale, Locale type
    types.ts                         # i18next declaration merging for TypeScript key-safety
    __tests__/
      locales.test.ts                # validates all locale files against en.json
  package.json                       # name: "@repo/i18n"
  tsconfig.json
```

### API surface

```ts
// index.ts exports
export const supportedLocales = ['en', 'vi'] as const;
export type Locale = typeof supportedLocales[number];
export const defaultLocale: Locale = 'en';

// Storage adapter interface ‚Äî implemented per platform
export interface I18nStorage {
  getItem(key: string): Promise<string | null>;
  setItem(key: string, value: string): Promise<void>;
}

export function createI18n(storage: I18nStorage): i18n;
```

### Translation namespaces

Flat structure to start ‚Äî one default namespace per locale file. Namespace splitting (e.g. `common`, `accounts`, `dashboard`) is deferred until files grow large enough to warrant it.

### TypeScript key-safety

`types.ts` uses i18next declaration merging so `t('missing.key')` is a compile error:

```ts
// types.ts
import enJson from './locales/en.json';

declare module 'i18next' {
  interface CustomTypeOptions {
    defaultNS: 'translation';
    resources: { translation: typeof enJson };
  }
}
```

### Locale validation test

`locales.test.ts` (Vitest):
- Loads `en.json` as the schema
- Auto-discovers every other `.json` in `locales/`
- Asserts each discovered file contains **all** keys from `en.json`
- Runs automatically when new locale files are added ‚Äî no changes to the test file needed

## 2. Web Integration ‚Äî `apps/web`

### Initialization

`app/i18n.ts` ‚Äî client module, calls `createI18n(localStorage)` once and exports the instance.  
`app/providers.tsx` ‚Äî wraps the app in `I18nextProvider` with the instance.

The `I18nStorage` adapter for web wraps `localStorage` with a Promise-based interface to match the shared contract.

### Language persistence flow

1. On app init: read saved locale from `localStorage` ‚Üí set as active language
2. On login: read `user.preferredLocale` from auth response ‚Üí override local value and re-save
3. On language change (Settings): call `i18n.changeLanguage(locale)` ‚Üí write to `localStorage` ‚Üí `PATCH /api/users/me` with `{ preferredLocale }`

### Language switcher

`components/LanguageSwitcher.tsx` ‚Äî uses Radix UI `<Select>` (already in the stack).  
Location: **Settings/profile screen only**.  
Each option shows native language name (e.g. "English", "Ti·∫øng Vi·ªát").

### Routing

No URL-based locale segments. Language is a user preference stored in state/storage, not encoded in the URL.

## 3. Mobile Integration ‚Äî `apps/mobile`

### Initialization

`lib/i18n.ts` ‚Äî calls `createI18n(asyncStorageAdapter)` where the adapter wraps `@react-native-async-storage/async-storage`.  
Initialized before the root layout mounts in `app/_layout.tsx`.

### Language persistence flow

Identical pattern to web ‚Äî replace `localStorage` with `AsyncStorage`.

### Language switcher

`components/LanguageSwitcher.tsx` ‚Äî renders a list of locales (native name).  
Location: **Settings/profile screen only**.  
Tapping an option calls `i18n.changeLanguage(locale)` and persists to `AsyncStorage`.

No special React Native plugins required ‚Äî `react-i18next` works out of the box in React Native.

## 4. API Changes

### Prisma schema

```prisma
model User {
  // existing fields ...
  preferredLocale String @default("en")
}
```

- Non-nullable, defaults to `"en"`
- Requires a new Prisma migration

### Auth response

`preferredLocale` is included in the JWT payload and/or user response object so the client can apply it on login without an extra round-trip.

### Update endpoint

`PATCH /api/users/me` accepts `{ preferredLocale: string }`.  
Validation: value must be in `supportedLocales` (imported from `@repo/i18n` ‚Äî single source of truth).

---

## 5. Adding a New Language

1. Add `packages/i18n/src/locales/<code>.json` with all keys from `en.json`
2. Add `<code>` to `supportedLocales` array in `packages/i18n/src/index.ts`
3. The locale test validates the new file automatically
4. The API validator accepts it automatically (reads `supportedLocales` at runtime)

## Out of Scope

- Locale-based URL routing (not needed ‚Äî preference is stored in user state)
- Right-to-left (RTL) layout support
- Number/date formatting (can be added via `i18next` plugins later)
- Language detection from browser/device locale (user sets explicitly in Settings)
