# i18n Design Spec — money-flow

**Date:** 2026-05-27  
**Status:** Approved  

## Summary

Add multi-language support (English + Vietnamese, extensible) to both the web (Next.js 16) and mobile (Expo) apps using `react-i18next`. Translation files live in a new shared `packages/i18n` package. Language preference is stored locally per device and synced to the user's profile on the API via the existing `language` field on `User`.

---

## 1. Shared Package — `packages/i18n`

New internal package `@repo/i18n`. Consumed only by web and mobile — the API does **not** depend on this package (see Section 4 for why).

### File structure

```
packages/i18n/
  src/
    locales/
      en.json                        # source of truth — all keys defined here
      vi.json                        # Vietnamese translations
    index.ts                         # exports: createI18n(), Locale type
    types.ts                         # i18next declaration merging for TypeScript key-safety
    __tests__/
      locales.test.ts                # validates all locale files against en.json
  package.json                       # name: "@repo/i18n"
  tsconfig.json
  vitest.config.ts
```

### `package.json`

```json
{
  "name": "@repo/i18n",
  "version": "0.0.0",
  "private": true,
  "exports": {
    ".": "./src/index.ts"
  },
  "scripts": {
    "test": "vitest run"
  },
  "dependencies": {
    "@repo/shared": "workspace:*"
  },
  "peerDependencies": {
    "i18next": "^23.0.0",
    "react": ">=18.0.0",
    "react-i18next": "^14.0.0"
  },
  "devDependencies": {
    "@repo/typescript-config": "workspace:*",
    "i18next": "^23.16.8",
    "react": "19.1.0",
    "react-i18next": "^14.1.3",
    "typescript": "5.9.2",
    "vitest": "^2.0.0"
  }
}
```

`react-i18next` and `i18next` are `peerDependencies` — each consuming app installs them directly. This prevents React from being pulled into the NestJS API's dependency graph if it ever needs to import from `@repo/i18n`. The `devDependencies` copy is for the Vitest tests only.

### `tsconfig.json`

Extends `@repo/typescript-config/react-library.json` (JSX support + `resolveJsonModule: true` for importing locale `.json` files):

```json
{
  "extends": "@repo/typescript-config/react-library.json",
  "compilerOptions": {
    "resolveJsonModule": true,
    "outDir": "dist"
  },
  "include": ["src"]
}
```

### API surface

```ts
// index.ts exports — Locale and supportedLocales are re-exported from @repo/shared/i18n
export type { Locale } from '@repo/shared/i18n';
export { supportedLocales, defaultLocale } from '@repo/shared/i18n';

// Storage adapter interface — implemented per platform
export interface I18nStorage {
  getItem(key: string): Promise<string | null>;
  setItem(key: string, value: string): Promise<void>;
}

// Returns a fully-initialized i18next instance synchronously.
// Uses initImmediate: false — safe because translations are bundled JSON (no network fetch).
// initialLocale: caller-resolved locale (from localStorage / AsyncStorage before first render).
export function createI18n(storage: I18nStorage, initialLocale?: Locale): i18n;
```

`Locale` is defined exactly once in `@repo/shared/i18n` and derived from `supportedLocales`. `@repo/i18n` re-exports it for convenience — there is no standalone union literal in `@repo/i18n`.

**Initialization is synchronous** via `initImmediate: false`. Translations are bundled JSON imports — no network fetch needed — so sync init is safe. Callers receive a ready instance with no need to await or guard with `isInitialized`.

### TypeScript key-safety

`types.ts` uses i18next declaration merging so `t('missing.key')` is a compile error:

```ts
import enJson from './locales/en.json';

declare module 'i18next' {
  interface CustomTypeOptions {
    defaultNS: 'translation';
    resources: { translation: typeof enJson };
  }
}
```

### Locale validation test

`locales.test.ts` (Vitest):
- Loads `en.json` as the schema
- Auto-discovers every other `.json` in `locales/`
- Asserts each discovered file contains **all** keys from `en.json` (recursive deep-key check)
- Adding a new locale file triggers the check automatically — no changes to the test file needed
- Runs via `pnpm --filter @repo/i18n test` or `turbo run test`

### Turbo pipeline

Add a `"test"` task to `turbo.json`:

```json
{
  "tasks": {
    "test": {
      "dependsOn": [],
      "outputs": []
    }
  }
}
```

## 2. `@repo/shared` — add `supportedLocales`

`supportedLocales` and `defaultLocale` live in `@repo/shared` (not `@repo/i18n`) so the NestJS API can import them without pulling in React. The API already depends on `@repo/shared`.

Add `packages/shared/src/i18n.ts`:

```ts
export const supportedLocales = ['en', 'vi'] as const;
export type Locale = typeof supportedLocales[number];
export const defaultLocale: Locale = 'en';
```

Add an export entry in `packages/shared/package.json`:

```json
{
  "exports": {
    "./currencies": "./src/currencies.ts",
    "./i18n": "./src/i18n.ts"
  }
}
```

`@repo/i18n/src/index.ts` re-exports `Locale` and `supportedLocales` from `@repo/shared/i18n` for convenience (so web/mobile only import from `@repo/i18n`).

## 3. Web Integration — `apps/web`

### Dependencies to add to `apps/web/package.json`

```
"@repo/i18n": "workspace:*"
"i18next": "^23.16.8"
"react-i18next": "^14.1.3"
```

### Initialization

`lib/i18n.ts` — a `"use client"` module. Guards against SSR with `typeof window !== 'undefined'` before accessing `localStorage`, identical to the existing pattern in `apps/web/lib/auth.ts`. On the server, the storage adapter returns `null` from `getItem`, so the default locale (`"en"`) is used during SSR. Because `initImmediate: false` makes `createI18n` synchronous, the instance is ready to use immediately as a module-level singleton.

`app/providers.tsx` — wraps the app in `I18nextProvider` with the instance.

### Metro / Metro is not relevant here (web uses Next.js webpack)

No bundler config changes needed for web.

### Language persistence flow

1. **On app init:** read saved locale from `localStorage` → set as active language (falls back to `"en"` if null)
2. **On login success:** read `language` from the auth response body → override local value, re-save to `localStorage`
3. **On register success:** same as login — read `language` from auth response and apply
4. **On language change (Settings):** call `i18n.changeLanguage(locale)` → write to `localStorage` → `PATCH /api/users/me` with `{ language }`

### Language switcher

`components/LanguageSwitcher.tsx` — uses Radix UI `<Select>` (already in the stack).  
Location: **Settings/profile screen only**.  
Each option shows native language name (e.g. "English", "Tiếng Việt").

### `html lang` attribute

`app/layout.tsx` currently has `<html lang="en">`. A small `HtmlLang` client component uses `useEffect` to call `document.documentElement.setAttribute('lang', i18n.language)` after hydration and on language changes.

**SSR limitation:** The server-rendered HTML will always have `lang="en"` because the user's locale preference is stored in `localStorage` (unavailable on the server, URL routing intentionally excluded). The `useEffect` updates `lang` on the client after hydration. This is an accepted trade-off: the lang attribute is cosmetic/accessibility metadata; it does not affect the rendered text (which is always correct because i18next uses the stored locale on the client).

### Routing

No URL-based locale segments. Language is a user preference stored in state/storage, not encoded in the URL.

## 4. Mobile Integration — `apps/mobile`

### Dependencies to add

```bash
# Use expo install to get the Expo SDK-compatible version
expo install @react-native-async-storage/async-storage
```

Also add to `apps/mobile/package.json`:

```
"@repo/i18n": "workspace:*"
"i18next": "^23.16.8"
"react-i18next": "^14.1.3"
```

Because `react-i18next` is a `peerDependency` of `@repo/i18n`, installing it directly in `apps/mobile` ensures Metro can resolve it from the app's own `node_modules` — no `watchFolders` changes needed.

### Metro config

`apps/mobile/metro.config.js` add `packages/i18n` to `watchFolders` so Metro picks up source changes:

```js
const { getDefaultConfig } = require('expo/metro-config');
const { withNativeWind } = require('nativewind/metro');
const path = require('path');

const config = getDefaultConfig(__dirname);

config.watchFolders = [
  path.resolve(__dirname, '../../packages/i18n'),
  path.resolve(__dirname, '../../packages/shared'),
];

module.exports = withNativeWind(config, { input: './global.css' });
```

The `withNativeWind` wrapper must be preserved — the existing `metro.config.js` already uses it.

### Initialization

`AsyncStorage.getItem` is genuinely async and cannot be awaited inside `createI18n`'s synchronous init. The correct pattern uses `SplashScreen` to block the first render until the locale is loaded:

**`lib/i18n.ts`**

```ts
import AsyncStorage from '@react-native-async-storage/async-storage';
import * as SplashScreen from 'expo-splash-screen';
import { createI18n } from '@repo/i18n';
import type { Locale } from '@repo/i18n';

SplashScreen.preventAutoHideAsync();

const asyncStorageAdapter = {
  getItem: (key: string) => AsyncStorage.getItem(key),
  setItem: (key: string, value: string) => AsyncStorage.setItem(key, value),
};

let i18nInstance: ReturnType<typeof createI18n> | null = null;

export async function initI18n() {
  const saved = await AsyncStorage.getItem('i18n-locale');
  i18nInstance = createI18n(asyncStorageAdapter, (saved as Locale) ?? 'en');
}

export function getI18n() {
  return i18nInstance!;
}
```

**`app/_layout.tsx`**

```ts
useEffect(() => {
  initI18n().then(() => SplashScreen.hideAsync());
}, []);
```

The `I18nStorage` adapter is used only for `setItem` (persisting language changes). The initial locale read is done by `initI18n()` before `createI18n` is called.

### Language persistence flow

Identical pattern to web — replace `localStorage` with `AsyncStorage`. Covers login, register, and in-app changes.

### Language switcher

`components/LanguageSwitcher.tsx` — renders a list of locales (native name).  
Location: **Settings/profile screen only**.  
Tapping an option calls `i18n.changeLanguage(locale)` and persists to `AsyncStorage`.

## 5. API Changes

### Prisma schema

The `User` model already has:

```prisma
language String @default("en")
```

**No new migration is needed.** No new field required.

### Auth response — login and register

Both `LoginUseCase` and `RegisterUseCase` currently return only `{ accessToken }`. Both must be updated to also return `language` from the user record in the response body (not the JWT payload — simpler, no token bloat):

```ts
return {
  accessToken: this.jwt.sign({ sub: user.id, email: user.email }),
  language: user.language,
};
```

The JWT payload stays as `{ sub, email }` — no change to `JwtStrategy`.

### Update endpoint — `PATCH /api/users/me`

This endpoint does not exist yet. Full additions required in order:

1. **`IUserRepository`** — add `update(id: string, data: Partial<UserEntity>): Promise<UserEntity>` to the interface (`src/users/domain/user.repository.interface.ts`)
2. **`UserRepository`** — add `update()` method using `prisma.user.update()` to the existing `src/users/infrastructure/user.repository.ts`
3. **`UpdateUserDto`** — new DTO (`src/users/application/dtos/update-user.dto.ts`) with `@IsIn(supportedLocales)` on the `language` field — import `supportedLocales` from `@repo/shared/i18n`
4. **`UpdateUserUseCase`** — new use case (`src/users/application/use-cases/update-user.use-case.ts`)
5. **`UsersController`** — new controller (`src/users/presentation/users.controller.ts`) with `@Patch('me')` guarded by `JwtAuthGuard`. Import the guard directly from its path (`../../auth/infrastructure/guards/jwt-auth.guard`) — do NOT import `AuthModule` (circular: `AuthModule` already imports `UsersModule`). Reads user ID from `req.user.userId` (per existing JWT strategy pattern)
6. **`UsersModule`** — wire the new controller and use case into the module (`PrismaModule` is `@Global()` so `PrismaService` is already available — no import needed)

`supportedLocales` from `@repo/shared/i18n` is the single source of truth for allowed locale values in the DTO validator.

## 6. Adding a New Language

1. Add `packages/i18n/src/locales/<code>.json` with all keys from `en.json`
2. Add `<code>` to `supportedLocales` array in `packages/shared/src/i18n.ts` — the `Locale` type derives automatically from it (no other file to update)
3. The locale test validates the new file automatically
4. The API `UpdateUserDto` validator accepts it automatically (reads `supportedLocales` at runtime)

## Out of Scope

- Locale-based URL routing
- Right-to-left (RTL) layout support
- Number/date formatting (can be added via `i18next` plugins later)
- Language detection from browser/device locale (user sets explicitly in Settings)
